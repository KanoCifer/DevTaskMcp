# devtask-grill

在正式 commit 到 plan/spec 之前，先敞开聊清楚：问题到底是什么、有哪些走法、各自代价和风险。

**核心：** 方案树展开 + 关键决策拷问。产出共享理解，**不落库 task**。

## 流程

### 1. 理解问题

先确认理解一致。如果有模糊之处或不明确的目标/约束，用 `AskUserQuestion` 确认（或引导澄清）：

- 有什么必须满足的非功能性要求？（性能、安全、兼容性…）
- 有 deadline 或资源限制吗？
- 这是一个新功能、重构、还是技术债偿还？

### 2. 探索上下文

涉及代码时，先查后问，不靠记忆：

- 模块名 → codegraph_explore / Read / grep
- "接 XXX 功能" → 找对应 endpoint / handler / service
- 涉及框架能力时优先查官方方案

**退出条件：** 了解当前代码现状、知道有哪些约束和可用的扩展点。避免在信息不足时讨论方案。

### 3. 方案树展开

列出 2-4 种可行方案，每种包含：

| 项目         | 说明                         |
| ------------ | ---------------------------- |
| **核心思路** | 一句话概括怎么干             |
| **代价**     | 代码改动量、新依赖、学习成本 |
| **风险**     | 哪个环节可能翻车             |
| **推荐度**   | 👑 推荐 / 👍 可行 / ⚠️ 有坑  |

用 `AskUserQuestion` 让用户选方向或表达偏好。一次只展开一层——选了大方向再深入细节，不一次性抛出所有分支。

### 4. 关键决策 Grilling

沿选定的方案树枝逐枝拷问，一次一问，等用户回答再出下一个：

1. **方案细节** — 这一步还有子选项吗？（e.g. 用 A 库还是 B 库）
2. **Hard-to-reverse 决策** — 这个选择后面还能改吗？确认用户理解后果
3. **脆弱假设** — 这个方案依赖什么为真？如果那个假设不成立怎么办？
4. **验收想象力** — 完工时怎么证明它工作？什么情况算"做完了"？

**原则：** 能从代码回答的不问；一次只问一个决策，不混。

### 5. 结论 + 下一步

总结讨论结果，推荐下一步：

```
## 讨论结论

**问题：** <一句话>
**选定方向：** <方案名 + 理由>
**未采纳选项：** <其他选项 + 未选理由>
**关键风险：** <仍需关注的点>

**下一步建议：**
- 复杂/跨层/多步骤 → `/devtask-plan` 出 spec + 子任务
- ≤5 文件/单层 → `/devtask-simple` 直接落库
```
